# Phase258C - Historical Fund Ownership Reconciliation & Opening Balance Recovery

## 1. Executive summary

Phase258C replaces the Phase258B blanket `ORDER_PAYMENT_ALLOCATION` non-balance rule with historical ownership resolution.

Current/future writer ownership remains unchanged:

- `OrderPaymentAllocationService.postAllocation()` does not create new `FUND:OPA:*` fund ledgers.
- `postFundLedgersFromAllocation()` remains fail-closed by default.
- Confirmed `DELIVERY_CASH_SUBMISSION` remains the current realized delivery remittance writer.

Historical canonical read ownership now distinguishes:

- `PROVEN_DUPLICATE`: OPA is excluded; DCS is counted.
- `LEGACY_ONLY`: OPA is preserved.
- `PARTIAL_OVERLAP`: OPA is not blindly removed; manual review is required.
- `AMBIGUOUS`: OPA is preserved and reported for review.

## 2. Root cause confirmed from code

Phase258B introduced `NON_BALANCE_SOURCE_TYPES = ORDER_PAYMENT_ALLOCATION` in `src/services/accounting/FundLedgerBalancePolicy.js`.

`FundBalanceReadService.fundLedgerCanonicalFilter()` used that policy before calculating:

- opening balance
- period inflow/outflow
- running balance
- dashboard balances
- finance report balances

Because opening balance is computed from canonical movements before `dateFrom`, blanket OPA exclusion removed legitimate historical legacy inflows and pulled every later balance wrong.

## 3. Why Phase258B blanket exclusion was historically unsafe

Phase258B correctly retired the future OPA fund writer, but it treated historical ownership and future writer ownership as the same concern.

That was unsafe because historical MK-Pro data can contain periods where `ORDER_PAYMENT_ALLOCATION` fund rows were the only fund movement evidence. Excluding every historical OPA by source type loses valid economic movement.

## 4. Call graph

### OPA writer

```text
AccountingCloseoutService
  -> OrderPaymentAllocationService.postAllocation()
    -> upsertAllocation()
    -> postArLedgersFromAllocation()
    -> fundLedgers = []
```

Legacy compatibility remains:

```text
postFundLedgersFromAllocation()
  -> throws ORDER_PAYMENT_ALLOCATION_FUND_POSTING_RETIRED
  -> only posts if allowLegacyOrderPaymentAllocationFundPosting === true
```

### DCS writer

```text
fundService.confirmDeliveryCashSubmission()
  -> postDeliveryRemittanceLine()
    -> postFundLedger()
      -> sourceType = DELIVERY_CASH_SUBMISSION
      -> sourceLineId = remittance line id
      -> date/accountingDate/remittanceDate = remittance date
      -> deliveryDate = delivery trip date
```

### Canonical read

```text
FundBalanceReadService
  -> HistoricalFundOwnershipPolicy
  -> opening/in/out/running balance
  -> fundController.listLedger
  -> FundDashboardReadService
  -> FinanceReportService
```

`FundSummaryQueryBuilder` applies the same ownership semantics in its aggregation pipeline.

## 5. Production audit methodology

Added `scripts/audit-historical-fund-ownership-reconciliation.js`.

The script:

- reads `fundLedgers`
- reads counts from `orderPaymentAllocations`
- reads counts from `deliveryCashSubmissions`
- classifies OPA rows by historical ownership
- writes JSON artifacts locally
- rejects `--apply` with `PHASE258C_AUDIT_READ_ONLY`
- does not update, delete, insert, bulkWrite, reverse, or create correction ledgers

The local workspace does not include production DB access, so the committed JSON files are explicitly marked as fixture/test evidence, not production DB evidence.

## 6. Matching identity hierarchy

The resolver uses:

1. `LEVEL_1_EXACT_IDENTITY`: direct technical identity overlap.
2. `LEVEL_2_STRONG_BUSINESS_IDENTITY`: one OPA and one DCS in the same delivery staff + delivery date + fund type group with exact amount and no competing group candidates.
3. `LEVEL_3_AGGREGATE_EXACT`: multiple OPA rows aggregate exactly to DCS amount in the same delivery staff + delivery date + fund type group.
4. `LEVEL_4_AMBIGUOUS`: partial overlap, missing identity, or insufficient evidence.

No amount-only match is used. No first candidate is selected.

## 7. Classification result

Fixture/test evidence:

- Exact duplicate: OPA 33,101,000 + DCS 33,101,000 -> canonical 33,101,000.
- Legacy-only OPA: OPA 20,000,000 without DCS -> canonical 20,000,000.
- Aggregate duplicate: 10,000,000 + 20,000,000 + 3,101,000 OPA vs one 33,101,000 DCS -> canonical 33,101,000.
- Partial overlap: OPA 33,101,000 vs DCS 30,000,000 -> manual review; OPA is not blindly removed.
- Ambiguous/missing identity -> OPA preserved.

Production classification must be generated by the read-only audit script.

## 8. Opening balance impact

Opening balance now uses Phase258C canonical ownership:

```text
openingBalance(D)
= sum(balance-affecting canonical movements before D)
```

Historical OPA before D is preserved unless DCS replacement is proven within the as-of scope. A delayed remittance test proves that a DCS on 2026-07-14 does not erase the OPA as-of 2026-07-13, but does supersede it for the 2026-07-14 canonical view when identity is strong.

## 12. Query/performance impact

Runtime balance queries still use aggregation, not N+1 client queries.

Phase258C adds a bounded `$setWindowFields` ownership resolution stage after early canonical/date/scope filtering. Partition keys are limited to OPA/DCS ownership groups; unrelated fund rows use row-specific partition keys to avoid one large blank partition.

Existing useful indexes:

- `idx_fund_ledgers_date_fund_direction`
- `idx_fund_ledgers_source_id_guard`
- `idx_fund_ledgers_reference_type_id`
- `idx_fund_ledgers_delivery_staff_date`
- `idx_fund_delivery_cash_submission_report`

No new production index was added.

## 13. Remaining manual review cases

Production rows classified as:

- `PARTIAL_OVERLAP`
- `AMBIGUOUS`
- missing delivery staff/date identity
- aggregate amount mismatch

must be reviewed before any correction or virtual adjustment is introduced.

## 14. Production deployment checklist

1. Run `node scripts/audit-historical-fund-ownership-reconciliation.js --opening-date=2026-07-14 --json`.
2. Review `PHASE258C_HISTORICAL_FUND_OWNERSHIP_AUDIT.json`.
3. Review `PHASE258C_FUND_OWNERSHIP_MANUAL_REVIEW.json`.
4. Open Quỹ tiền -> Sổ quỹ for 2026-07-14.
5. Compare cash/bank opening, inflow, outflow, closing.
6. Confirm running balance line invariant.
7. Compare Sổ quỹ, Tổng quan quỹ, Sổ quỹ tổng hợp, and Report Center finance.
8. Verify legacy-only OPA still affects balance.
9. Verify proven duplicate OPA + DCS affects balance once.
10. Create a new closeout and confirm no `FUND:OPA:*` is produced.
11. Confirm delivery remittance and verify exactly one DCS fund movement per confirmed line.

## 15. Rollback strategy

Rollback is code-only:

- restore Phase258B policy if emergency double-count risk is discovered
- do not delete or mutate historical fundLedgers
- do not bulk reverse OPA/DCS
- keep audit artifacts for reconciliation comparison

If rollback is used, rerun the audit script first and preserve the output before redeploying.
